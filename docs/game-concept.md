# ARBORITO: KNOWLEDGE INJECTION PROTOCOL (ZERO-TOUCH)

Internal design notes (not required to run or modify the app). For the product roadmap see `ROADMAP.md` in the repo root.

## 1. The Vision: Decoupled Curriculum and Narrative

Arborito's vision for educational games is to separate **"What is learned"** from **"How it's played"**.

*   **The Teacher (Arborito Studio):** Writes lessons in plain text or Markdown. Knows nothing about programming or games.
*   **The Game Developer:** Creates narrative containers (scenes, dialogues) without hardcoding questions.
*   **The Game Engine (Runtime):** Downloads raw content from Arborito and uses a local LLM (native llama.cpp in the desktop build, wllama/WebAssembly in the browser) to "understand" the lesson and generate game mechanics on the fly.

---

## 2. "The Architect and The Referee" Architecture

Since Arborito trees deliver content as-is (with HTML, Markdown, and noise that the author wrote), intelligence is moved to the client (the game) in two steps:

### Phase A: The Architect (Level Load)

When the game loads an Arborito module, it doesn't display the text. It sends it to an LLM with the role of "Architect".

*   **Input:** Raw text with HTML/Markdown tags (`<p>The <strong>photosynthesis</strong>...</p>`).
*   **Process:** The LLM cleans the text and extracts a logical structure (JSON) in memory.
*   **Output:** `{"key_concepts": ["Chlorophyll", "Sunlight"], "traps": ["Darkness"]}`.

### Phase B: The Referee (Gameplay)

During the game, each player interaction is sent to the LLM with the role of "Referee".

*   **Input:** Player action + Rules generated by the Architect.
*   **Process:** The Referee decides if the action demonstrates an understanding of the extracted concepts.
*   **Output:** Reactive narrative ("The guard lets you pass because you explained photosynthesis well.").

---

## 3. Advantages of the Zero-Touch Model

1.  **Total Compatibility:** Works with any existing Arborito tree, without the author needing to use special markers.
2.  **Zero Maintenance:** If the teacher edits the lesson text in the published tree, the "Architect" can regenerate rules the next time the game is played.
3.  **Resilience:** The game itself is responsible for "purifying" the data (removing HTML, formatting), making the Arborito backend as simple as a static file server.

## 5. Where the tree comes from

Cartridges and Python apps never bundle a curriculum themselves — they ask the host for lessons. Who chooses the tree depends on the surface:

- **Browser cartridges (Arcade):** the *user* picks the tree by opening it in Arborito and launching the game from a module. The cartridge sees only the resulting playlist.
- **Python SDK games (offline / Pygame / CLI):** the *developer* picks the tree at startup. Three options:
  - A pinned exported file (`Arborito.from_arborito("course.arborito")`) — offline, reproducible, ships with the game.
  - A static `data/` folder (`Arborito.from_static_data("/path/to/data")`) — same data, different layout.
  - A public **share code** that fetches the tree live from Nostr (`Arborito.from_share_code("ABCD-EF23")`), optionally with `arb.subscribe(...)` to react to new versions published by the author.

The full contract and the exact recipe to ship a game pinned to a specific tree (or to follow a live one) live in [`sdk-spec.md` → "Selecting the curriculum source"](./sdk-spec.md#selecting-the-curriculum-source-which-tree-powers-your-game).
